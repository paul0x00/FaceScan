# FaceScan 项目接手指南

本文档面向新接手项目的开发者，重点说明代码框架、模块职责、核心业务流和常见修改入口。更详细的安装命令可参考根目录 `README.md`。

## 1. 项目定位

FaceScan 是一个本地化面部扫描工作站原型，采用 C++ 本地后端 + Vue 前端。系统围绕以下流程展开：

1. 登录本地工作站。
2. 创建或编辑患者档案。
3. 创建订单并进行多视角相机采集。
4. 基于采集图片生成彩色 PLY 点云。
5. 在前端查看点云。
6. 本地打包交付，云端上传接口目前是占位实现。

开发期由 Vite 前端服务把 `/api` 代理到 C++ 后端：

```text
浏览器 / Vue 应用
  -> frontend/src/api/client.ts
  -> Vite proxy /api
  -> C++ Boost.Beast HTTP server
  -> App::handle 手写路由
  -> Database / CameraManager / ColorPointCloudSdk / 文件系统
```

## 2. 快速启动

Windows 推荐从仓库根目录执行：

```powershell
.\scripts\windows\dev.cmd
```

常用变体：

```powershell
.\scripts\windows\dev.cmd -InstallTools
.\scripts\windows\dev.cmd -CleanBuild
.\scripts\windows\dev.cmd -NoFrontend
.\scripts\windows\stop-dev.cmd
```

默认地址：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:8080`
- 开发登录：`admin / admin`

后端单独构建：

```powershell
cmake -S backend --preset windows
cmake --build backend/build --config Release
.\backend\build\Release\facescan_backend.exe 8080
```

前端单独启动：

```powershell
cd frontend
npm install
npm run dev -- --host 127.0.0.1
```

## 4. 后端结构

后端入口很薄，主要逻辑集中在 `App`：

```text
backend/src/
├── main.cpp                         # 进程入口，调用 runBackend
├── server/http_server.*             # Boost.Asio/Beast 监听、连接和请求循环
├── api/app.*                        # API 路由和业务编排中心
├── database/database.*              # SQLite 建表、迁移、CRUD、索引同步
├── camera/camera_manager.*          # 相机抽象、mock 相机、Orbbec 相机
├── algorithm/color_point_cloud_sdk.*# 采集图 -> PLY 点云
├── config/app_config.*              # app.json 读取、默认配置、保存
└── common/                          # 模型、JSON、时间、文件工具
```

### 4.1 启动链路

1. `backend/src/main.cpp` 调用 `facescan::runBackend(argc, argv)`。
2. `server/http_server.cpp` 读取 `AppConfig`，决定监听端口。
3. `runBackend` 创建 `std::shared_ptr<App>`。
4. 每个 TCP 连接启动一个 detached 线程。
5. 每个 HTTP 请求交给 `App::handle`。

注意：后端当前是“每连接一线程”的简单模型，数据库访问靠 `Database` 内部 mutex 串行化。

### 4.2 App 编排层

`backend/src/api/app.cpp` 是后端最重要的文件，负责：

- 解析 path、method、query、body。
- 以手写 `if` 阶梯匹配路由。
- 组织数据库、相机、点云、文件系统调用。
- 直接拼接 JSON 响应。
- 做文件访问边界校验，例如图片和 PLY 必须位于 `imageRoot` 下。

新增接口通常需要改三个地方：

1. `backend/src/api/app.cpp` 增加路由和处理函数。
2. `frontend/src/api/client.ts` 增加类型化 API 方法。
3. `frontend/src/types/index.ts` 补齐请求或响应类型。

### 4.3 数据库层

数据库文件固定保存到：

```text
backend/data/db/facescan.sqlite3
```

即使配置文件中有 `database` 字段，当前代码会在 `app_config.cpp` 中用固定路径覆盖它，避免数据库跟着用户数据目录迁移。

核心表：

- `patient`：患者基础资料、患者编号、下一个订单序号。
- `orders`：订单编号、状态、所属患者、创建时间。
- `scan_result`：订单采集图片、点云 PLY、预览图、创建时间。

关键约束：

- `orders(patient_id, order_no)` 有唯一索引，同一患者下订单号不能重复。
- `Database` 构造时会建表、补列、清理孤儿记录、去重订单。
- `syncDataRoot` 会扫描图片根目录，把真实文件反向同步到数据库索引。

### 4.4 相机层

相机通过 `ICameraDevice` 抽象，`CameraManager` 对 API 层暴露统一能力：

- `start` / `stop`
- `controls` / `updateControls`
- `frameImage(view)`
- `capture(orderFolder, orderName)`

支持两类设备：

- `MockCameraDevice`：默认模式，无硬件也能跑完整流程，会生成模拟预览和采集文件。
- `OrbbecCameraDevice`：仅在 CMake 检测到 Orbbec SDK 时编译，配置 `cameraMode` 为 `orbbec` 或 `gemini215` 时使用。

Orbbec SDK 路径由 `backend/CMakeLists.txt` 根据平台自动查找：

```text
backend/3rdParty/orbbec/windows
backend/3rdParty/orbbec/macos
backend/3rdParty/orbbec/linux
```

也可以通过 CMake 参数覆盖：

```bash
-DFACESCAN_ORBBEC_SDK_DIR=...
```

### 4.5 点云层

`ColorPointCloudSdk` 负责把采集图转换成彩色 PLY 点云：

```text
latestImagePathsForOrder
  -> ColorPointCloudSdk::reconstruct
  -> 写出 <订单号>.ply
  -> buildPointCloudPreviewSvg
  -> Database::setPointCloudResult
```

当前默认采样参数在前端调用中设置为 `columns: 96, rows: 72`，后端默认值是 `80 x 60`。如果要调点云密度，优先看：

- `frontend/src/api/client.ts` 的 `reconstructOrder`
- `backend/src/algorithm/color_point_cloud_sdk.*`
- `backend/src/api/app.cpp` 的 `reconstructOrder`

```text
frontend/src/
├── main.ts                  # Vue 装配入口
├── App.vue                  # 应用外壳
├── router/index.ts          # 页面路由和登录守卫
├── api/client.ts            # Axios API 封装
├── stores/patients.ts       # 患者列表共享状态
├── types/index.ts           # 前端类型定义
├── components/              # StepHeader、FormPanel、LogoMark 等通用组件
├── views/                   # 业务页面
└── styles.css               # 全局样式
```

### 5.1 路由

页面路由如下：

- `/login`：本地登录。
- `/browse`：患者、订单、扫描记录浏览和入口页。
- `/basic/:id?`：新建或编辑患者资料。
- `/shoot/:id`：相机预览、调参、同步采集。
- `/pointcloud/:id`：点云加载和 vtk.js 渲染。
- `/send/:id`：本地打包和预留上传。
- `/settings`：数据目录设置。

除 `/login` 外，路由守卫只检查 `localStorage` 中是否存在 `facescan.token`。后端目前返回固定开发 token，没有真正鉴权。

### 5.2 API 客户端

`frontend/src/api/client.ts` 是前后端接口的集中入口：

- `baseURL` 是 `/api`。
- 请求拦截器把 `facescan.token` 放入 `Authorization: Bearer ...`。
- 所有页面都应该通过这里调用后端，避免在页面里散落 axios 逻辑。

新增页面功能时，优先在这里补一个语义化函数，再在 view 中调用。

### 5.3 主要页面职责

- `BrowseView.vue`：加载患者列表、筛选日期、选择患者、查看订单、创建/删除订单、打开订单目录。
- `BasicInfoView.vue`：新建/编辑患者，保存后跳到拍摄页。
- `ShootView.vue`：启动相机、轮询四路预览、读取/写入相机参数、同步采集、触发点云生成。
- `PointCloudView.vue`：加载 PLY 文件，用 `vtk.js` 渲染点云，支持重置视角、纹理开关和六视图。
- `SendView.vue`：读取扫描记录，执行本地 ZIP 打包，调用占位上传接口。
- `SettingsView.vue`：读取、校验、保存数据目录，并触发目录迁移。

## 6. 核心业务流程

### 6.1 登录

```text
LoginView
  -> login(username, password)
  -> POST /api/login
  -> localStorage.facescan.token = dev-token
  -> 跳转 /browse
```

后端目前不校验用户名密码，直接返回开发 token。

### 6.2 建档和订单

```text
BasicInfoView submit
  -> createPatient / updatePatient
  -> POST /api/patients 或 PUT /api/patients/{id}
  -> Database 写 patient
  -> 新建患者时自动创建首个 order
  -> 跳转 /shoot/{patientId}?orderId={orderId}
```

订单号由数据库层生成。创建新订单时调用：

```text
POST /api/orders { patientId }
```

### 6.3 采集

```text
ShootView onMounted
  -> POST /api/camera/start
  -> GET /api/camera/controls
  -> 每路 img 轮询 /api/camera/frame?view=...

点击同步拍摄
  -> POST /api/camera/capture { patientId, orderId }
  -> CameraManager::capture
  -> 写入订单目录
  -> Database::replaceOrderCapture
```

采集文件位于：

```text
<imageRoot>/<患者编号-姓名>/<订单编号>/
```

mock 模式会生成模拟文件；真实设备模式会保存 Orbbec 同步采集结果。

### 6.4 点云生成和查看

```text
ShootView 下一步
  -> POST /api/orders/{orderId}/reconstruct
  -> 读取订单最近采集图
  -> ColorPointCloudSdk::reconstruct
  -> 写 PLY 和 SVG 预览
  -> Database::setPointCloudResult
  -> 跳转 /pointcloud/{patientId}?orderId=...&plyPath=...

PointCloudView
  -> GET /api/pointcloud/file?path=...
  -> vtkPLYReader 加载 PLY
  -> vtk.js 渲染点云
```

### 6.5 交付

```text
SendView
  -> GET /api/patients/{id}/scans
  -> POST /api/export/package { patientId, orderId }
  -> 后端收集图片、PLY、预览图
  -> 写 <订单号>_package.zip
```

`POST /api/export/upload` 目前只返回成功文案，不做真实上传。

### 6.6 设置和数据迁移

```text
SettingsView
  -> GET /api/settings
  -> POST /api/settings/validate-data-root
  -> PUT /api/settings
```

保存新数据目录时，后端会：

1. 校验路径不为空、不包含 `..`、不能指向数据库目录或后端根目录。
2. 如路径变化，移动患者目录。
3. 批量替换数据库中已保存路径前缀。
4. 写入 `backend/config/app.json`。
5. 更新相机保存根目录。

## 8. 配置和数据落盘

配置文件查找顺序：

```text
config/app.json
config/app.example.json
backend/config/app.json
backend/config/app.example.json
```

示例配置：

```json
{
  "backendPort": 8080,
  "database": "data/db/facescan.sqlite3",
  "imageRoot": "data/images",
  "cameraMode": "mock"
}
```

当前重要约定：

- `backendPort` 可被命令行第一个参数覆盖。
- `database` 字段保留展示，实际数据库路径固定为 `backend/data/db/facescan.sqlite3`。
- `imageRoot` 是患者图片、点云、预览图和打包文件的根目录。
- `cameraMode=mock` 时无需硬件。
- `cameraMode=orbbec` 或 `gemini215` 时需要 CMake 成功检测到 Orbbec SDK。

## 9. 测试

后端测试位于 `backend/tests/`，覆盖 API、数据库、相机、JSON 工具和点云算法：

```powershell
cmake -S backend --preset windows
cmake --build backend/build --config Release
ctest --test-dir backend/build -C Release --output-on-failure
```

前端生产构建会先执行类型检查：

```powershell
cd frontend
npm run build
```

修改建议：

- 改数据库行为：补或更新 `backend/tests/database_test.cpp`。
- 改 API 路由：补或更新 `backend/tests/app_test.cpp`。
- 改相机抽象：补或更新 `backend/tests/camera_manager_test.cpp`。
- 改点云算法：补或更新 `backend/tests/color_point_cloud_sdk_test.cpp`。
- 改前端类型/API：至少跑 `npm run build`。

## 11. 接手时最需要注意的点

1. `App::handle` 是单文件手写路由，没有独立路由框架，路径解析以字符串截取为主。
2. JSON 解析和响应拼接是项目自写工具，不是完整 JSON 库，复杂结构要特别小心转义和缺省值。
3. 数据库路径固定，数据图片根目录可配置，两者不要混淆。
4. 数据库索引会反扫 `imageRoot`，所以直接手工放入符合命名规则的患者/订单目录，也可能被同步进数据库。
5. 文件读取接口会要求文件位于 `imageRoot` 下，调试图片或 PLY 加载失败时先看路径前缀。
6. mock 相机是默认可跑通路径，真实 Orbbec 依赖 CMake 检测和运行时 DLL/dylib。
7. 前端点云查看依赖 `vtk.js` 读取后端返回的文本 PLY。
8. 云端上传还未实现真实业务，只是占位成功响应。
9. Windows 启动脚本会把后端和前端作为隐藏进程启动，日志在 `Log/`。
10. 代码中已有中文注释，但部分文件在当前终端可能显示乱码；编辑时保持 UTF-8。

## 12. 建议的新功能开发流程

1. 先确认需求属于患者、订单、采集、点云、交付、设置中的哪一类。
2. 从对应 view 找到用户入口。
3. 通过 `frontend/src/api/client.ts` 找到前端调用。
4. 到 `backend/src/api/app.cpp` 找到后端路由。
5. 如果涉及持久化，再进入 `Database` 和 `models.hpp`。
6. 如果涉及文件，确认路径必须在 `imageRoot` 下。
7. 补测试或至少运行对应构建命令。
8. 用 mock 模式先跑通，再接真实设备。
